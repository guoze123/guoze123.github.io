  1. 1. 代码风格
    1. 1.1. 结构
      1. 1.1.1. 缩进
        1. 1.1.1.1. [强制] 使用 2 个空格做为一个缩进层级
        2. 1.1.1.2. [强制] switch 下的 case 和 default 必须增加一个缩进层级
      2. 1.1.2. 空格
        1. 1.1.2.1. [强制] 二元运算符两侧必须有一个空格，一元运算符与操作对象之间不允许有空格
        2. 1.1.2.2. [强制] 用作代码块起始的左花括号 &#123; 前必须有一个空格
        3. 1.1.2.3. [强制] if / else / for / while / function / switch / do / try / catch / finally 关键字后，必须有一个空格
        4. 1.1.2.4. [强制] 在对象创建时，属性中的 : 之后必须有空格，: 之前不允许有空格
        5. 1.1.2.5. [强制] 函数声明、具名函数表达式、函数调用中，函数名和 ( 之间不允许有空格
        6. 1.1.2.6. [强制] , 和 ; 前不允许有空格。如果不位于行尾，, 和 ; 后必须跟一个空格
        7. 1.1.2.7. [强制] 在函数调用、函数声明、括号表达式、属性访问、if / for / while / switch / catch 等语句中，() 和 [] 内紧贴括号部分不允许有空格
        8. 1.1.2.8. [强制] 单行声明的数组与对象，如果包含元素，&#123;&#125; 和 [] 内紧贴括号部分不允许包含空格
        9. 1.1.2.9. [强制] 行尾不得有多余的空格
      3. 1.1.3. 换行
        1. 1.1.3.1. [强制] 每个独立语句结束后必须换行
        2. 1.1.3.2. [强制] 每行不得超过 120 个字符
        3. 1.1.3.3. [强制] 运算符处换行时，运算符必须在新行的行首
        4. 1.1.3.4. [强制] 在函数声明、函数表达式、函数调用、对象创建、数组创建、for 语句等场景中，不允许在 , 或 ; 前换行
        5. 1.1.3.5. [建议] 不同行为或逻辑的语句集，使用空行隔开，更易阅读
        6. 1.1.3.6. [建议] 在语句的行长度超过 120 时，根据逻辑条件合理缩进
        7. 1.1.3.7. [建议] 对于 if...else...、try...catch...finally 等语句，推荐使用在 &#125; 号后添加一个换行 的风格，使代码层次结构更清晰，阅读性更好
      4. 1.1.4. 语句
        1. 1.1.4.1. [强制] 不得省略语句结束的分号
        2. 1.1.4.2. [强制] 在 if / else / for / do / while 语句中，即使只有一行，也不得省略块 &#123;...&#125;
        3. 1.1.4.3. [强制] 函数定义结束不允许添加分号
        4. 1.1.4.4. [强制] IIFE 必须在函数表达式外添加 (，非 IIFE 不得在函数表达式外添加 (
    2. 1.2. 命名
      1. 1.2.0.1. [强制] 变量 使用 Camel命名法
      2. 1.2.0.2. [强制] 常量 使用 全部字母大写，单词间下划线分隔 的命名方式
      3. 1.2.0.3. [强制] 函数 使用 Camel命名法
      4. 1.2.0.4. [强制] 函数的 参数 使用 Camel命名法
      5. 1.2.0.5. [强制] 类 使用 Pascal命名法
      6. 1.2.0.6. [强制] 类的 方法 / 属性 使用 Camel命名法
      7. 1.2.0.7. [强制] 枚举变量 使用 Pascal命名法，枚举的属性 使用 全部字母大写，单词间下划线分隔 的命名方式
      8. 1.2.0.8. [强制] 命名空间 使用 Camel命名法
      9. 1.2.0.9. [强制] 由多个单词组成的缩写词，在命名中，根据当前命名法和出现的位置，所有字母的大小写与首字母的大小写保持一致
      10. 1.2.0.10. [强制] 类名 使用 名词
      11. 1.2.0.11. [建议] 函数名 使用 动宾短语
      12. 1.2.0.12. [建议] boolean 类型的变量使用 is 或 has 开头
      13. 1.2.0.13. [建议] Promise对象 用 动宾短语的进行时 表达
  2. 1.3. 注释
    1. 1.3.1. 单行注释
      1. 1.3.1.1. [强制] 必须独占一行。// 后跟一个空格，缩进与下一行被注释说明的代码一致
    2. 1.3.2. 多行注释
      1. 1.3.2.1. [建议] 避免使用 /*...*/ 这样的多行注释。有多行注释内容时，使用多个单行注释
    3. 1.3.3. 文档化注释
      1. 1.3.3.1. [强制] 为了便于代码阅读和自文档化，以下内容必须包含以 /**...*/ 形式的块注释中
      2. 1.3.3.2. [强制] 文档注释前必须空一行
      3. 1.3.3.3. [建议] 自文档化的文档说明 what，而不是 how
    4. 1.3.4. 类型定义
      1. 1.3.4.1. [强制] 类型定义都是以 &#123; 开始, 以 &#125; 结束
      2. 1.3.4.2. [强制] 对于基本类型 {string}, {number}, {boolean}，首字母必须小写
    5. 1.3.5. 文件注释
      1. 1.3.5.1. [强制] 文件顶部必须包含文件注释，用 @file 标识文件说明
      2. 1.3.5.2. [建议] 文件注释中可以用 @author 标识开发者信息
    6. 1.3.6. 命名空间注释
      1. 1.3.6.1. [建议] 命名空间使用 @namespace 标识
    7. 1.3.7. 类注释
      1. 1.3.7.1. [建议] 使用 @class 标记类或构造函数
      2. 1.3.7.2. [建议] 使用 @extends 标记类的继承信息
      3. 1.3.7.3. [强制] 使用包装方式扩展类成员时， 必须通过 @lends 进行重新指向
      4. 1.3.7.4. [强制] 类的属性或方法等成员信息不是 public 的，应使用 @protected 或 @private 标识可访问性
    8. 1.3.8. 函数/方法注释
      1. 1.3.8.1. [强制] 函数/方法注释必须包含函数说明，有参数和返回值时必须使用注释标识
      2. 1.3.8.2. [强制] 参数和返回值注释必须包含类型信息，且不允许省略参数的说明
      3. 1.3.8.3. [建议] 当函数是内部函数，外部不可访问时，可以使用 @inner 标识
      4. 1.3.8.4. [强制] 对 Object 中各项的描述， 必须使用 @param 标识
      5. 1.3.8.5. [建议] 重写父类方法时， 应当添加 @override 标识。如果重写的形参个数、类型、顺序和返回值类型均未发生变化，可省略 @param、@return，仅用 @override 标识，否则仍应作完整注释
    9. 1.3.9. 事件注释
      1. 1.3.9.1. [强制] 必须使用 @event 标识事件，事件参数的标识与方法描述的参数标识相同
      2. 1.3.9.2. [强制] 在会广播事件的函数前使用 @fires 标识广播的事件，在广播事件代码前使用 @event 标识事件
      3. 1.3.9.3. [建议] 对于事件对象的注释，使用 @param 标识，生成文档时可读性更好
    10. 1.3.10. 常量注释
      1. 1.3.10.1. [强制] 常量必须使用 @const 标记，并包含说明和类型信息
    11. 1.3.11. 复杂类型注释
      1. 1.3.11.1. [建议] 对于类型未定义的复杂结构的注释，可以使用 @typedef 标识来定义
    12. 1.3.12.
    13. 1.3.13. 细节注释
    14. 1.3.14. [建议] 细节注释遵循单行注释的格式。说明必须换行时，每行是一个单行注释的起始
      1. 1.3.14.1. [强制] 有时我们会使用一些特殊标记进行说明。特殊标记必须使用单行注释的形式。下面列举了一些常用标记
  • 2. 语言特性
    1. 2.1. 变量
      1. 2.1.1. [强制] 变量、函数在使用前必须先定义
      2. 2.1.2. [强制] 使用 let 和 const 定义变量，不使用 var
      3. 2.1.3. [强制] 变量必须 即用即声明，不得在函数或其它形式的代码块起始位置统一声明所有变量
    2. 2.2. 解构
      1. 2.2.1. [强制] 不要使用3层及以上的解构
      2. 2.2.2. [建议] 使用解构减少中间变量
      3. 2.2.3. [强制] 仅定义一个变量时不允许使用解构
      4. 2.2.4. [强制] 如果不节省编写时产生的中间变量，解构表达式 = 号右边不允许是 ObjectLiteral 和 ArrayLiteral
      5. 2.2.5. [强制] 使用剩余运算符时，剩余运算符之前的所有元素必需具名
    3. 2.3. 条件
      1. 2.3.0.1. [强制] 在 Equality Expression 中使用类型严格的 ===。仅当判断 null 或 undefined 时，允许使用 == null
      2. 2.3.0.2. [建议] 尽可能使用简洁的表达式
      3. 2.3.0.3. [建议] 按执行频率排列分支的顺序
      4. 2.3.0.4. [建议] 对于相同变量或表达式的多值条件，用 switch 代替 if
      5. 2.3.0.5. [建议] 如果函数或全局中的 else 块后没有任何语句，可以删除 else
  • 2.4. 循环
    1. 2.4.0.1. [建议] 不要在循环体中包含函数表达式，事先将函数提取到循环体外
    2. 2.4.0.2. [建议] 对循环内多次使用的不变值，在循环外用变量缓存
    3. 2.4.0.3. [建议] 对有序集合进行遍历时，缓存 length
    4. 2.4.0.4. [建议] 对有序集合进行顺序无关的遍历时，使用逆序遍历
  • 2.5. 类型
    1. 2.5.1. 类型检测
      1. 2.5.1.1. [建议] 类型检测优先使用 typeof。对象类型检测使用 instanceof。null 或 undefined 的检测使用 == null
    2. 2.5.2. 类型转换
      1. 2.5.2.1. [建议] 转换成 string 时，使用 + ''
      2. 2.5.2.2. [建议] 转换成 number 时，通常使用 +
      3. 2.5.2.3. [建议] string 转换成 number，要转换的字符串结尾包含非数字并期望忽略时，使用 parseInt
      4. 2.5.2.4. [强制] 使用 parseInt 时，必须指定进制
      5. 2.5.2.5. [建议] 转换成 boolean 时，使用 !!
      6. 2.5.2.6. [建议] number 去除小数点，使用 Math.floor / Math.round / Math.ceil，不使用 parseInt
  • 2.6. 3.3 模板字符串
    1. 2.6.1. [强制] 字符串内变量替换时，不要使用 2 次及以上的函数调用
  • 2.7. 对象
    1. 2.7.1. [强制] 使用对象字面量 &#123;&#125; 创建新 Object
    2. 2.7.2. [建议] 对象创建时，如果一个对象的所有 属性 均可以不添加引号，建议所有 属性 不添加引号
    3. 2.7.3. [建议] 对象创建时，如果任何一个 属性 需要添加引号，则所有 属性 建议添加 '
    4. 2.7.4. [强制] 不允许修改和扩展任何原生对象和宿主对象的原型
    5. 2.7.5. [建议] 属性访问时，尽量使用 .
    6. 2.7.6. [建议] for in 遍历对象时, 使用 hasOwnProperty 过滤掉原型中的属性
    7. 2.7.7. [建议] 定义对象时，如果所有键均指向同名变量，则所有键都使用缩写；如果有一个键无法指向同名变量，则所有键都不使用缩写
    8. 2.7.8. [强制] 定义方法时使用 MethodDefinition 语法，不使用 PropertyName: FunctionExpression 语法
    9. 2.7.9. [建议] 使用 Object.keys 或 Object.entries 进行对象遍历
    10. 2.7.10. [建议] 定义对象的方法不应使用箭头函数
    11. 2.7.11. [建议] 尽量使用计算属性键在一个完整的字面量中完整地定义一个对象，避免对象定义后直接增加对象属性
  • 2.8. 数组
    1. 2.8.0.1. [强制] 使用数组字面量 [] 创建新数组，除非想要创建的是指定长度的数组
    2. 2.8.0.2. [强制] 遍历数组不使用 for in
    3. 2.8.0.3. [建议] 不因为性能的原因自己实现数组排序功能，尽量使用数组的 sort 方法
    4. 2.8.0.4. [建议] 清空数组使用 .length = 0
  • 2.9. 函数
    1. 2.9.1. 函数长度
      1. 2.9.1.1. [建议] 一个函数的长度控制在 50 行以内
    2. 2.9.2. 参数设计
      1. 2.9.2.1. [建议] 一个函数的参数控制在 6 个以内
      2. 2.9.2.2. [建议] 通过 options 参数传递非数据输入型参数
    3. 2.9.3. 闭包
      1. 2.9.3.1. [建议] 在适当的时候将闭包内大对象置为 null
      2. 2.9.3.2. [建议] 使用 IIFE 避免 Lift 效应
    4. 2.9.4. 空函数
      1. 2.9.4.1. [建议] 空函数不使用 new Function() 的形式
      2. 2.9.4.2. [建议] 对于性能有高要求的场合，建议存在一个空函数的常量，供多处使用共享
    5. 2.9.5. [建议] 使用变量默认语法代替基于条件判断的默认值声明
    6. 2.9.6. [强制] 不要使用 arguments 对象，应使用 ...args 代替
  • 2.10. 类
    1. 2.10.1. [强制] 使用 class 关键字定义一个类
    2. 2.10.2. [强制] 使用 super 访问父类成员，而非父类的 prototype
  • 2.11. 3.8 模块
    1. 2.11.1. [强制] export 与内容定义放在一起
    2. 2.11.2. [建议] 相互之间无关联的内容使用命名导出
    3. 2.11.3. [强制] 所有 import 语句写在模块开始处
  • 2.12. 集合
    1. 2.12.1. [建议] 对数组进行连接操作时，使用数组展开语法
    2. 2.12.2. [建议] 不要使用数组展开进行数组的复制操作
    3. 2.12.3. [建议] 尽可能使用 for .. of 进行遍历
    4. 2.12.4. [强制] 当键值有可能不是字符串时，必须使用 Map；当元素有可能不是字符串时，必须使用 Set
    5. 2.12.5. [建议] 需要一个不可重复的集合时，应使用 Set
    6. 2.12.6. [建议] 当需要遍历功能时，使用 Map 和 Set
    7. 2.12.7. [建议] 程序运行过程中有添加或移除元素的操作时，使用 Map 和 Set
  • 2.13. 异步
    1. 2.13.1. [强制] 回调函数的嵌套不得超过3层
    2. 2.13.2. [建议] 使用 Promise 代替 callback
    3. 2.13.3. [强制] 使用标准的 Promise API
    4. 2.13.4. [强制] 不允许直接扩展 Promise 对象的 prototype
    5. 2.13.5. [强制] 不得为了编写的方便，将可以并行的IO过程串行化
    6. 2.13.6. [建议] 使用 async/await 代替 generator + co
  • 2.14. 动态特性
    1. 2.14.1. 3.10.1 eval
      1. 2.14.1.1. [强制] 避免使用直接 eval 函数
      2. 2.14.1.2. [建议] 尽量避免使用 eval 函数
    2. 2.14.2. 3.10.2 动态执行代码
      1. 2.14.2.1. [建议] 使用 new Function 执行动态代码
    3. 2.14.3. 3.10.3 with
      1. 2.14.3.1. [建议] 尽量不要使用 with
    4. 2.14.4. 3.10.4 delete
      1. 2.14.4.1. [建议] 减少 delete 的使用
      2. 2.14.4.2. [建议] 处理 delete 可能产生的异常
    5. 2.14.5. 3.10.5 对象属性
      1. 2.14.5.1. [建议] 避免修改外部传入的对象
      2. 2.14.5.2. [建议] 具备强类型的设计
  • 3. 浏览器环境
    1. 3.1. DOM
      1. 3.1.1. 元素获取
        1. 3.1.1.1. [建议] 对于单个元素，尽可能使用 document.getElementById 获取，避免使用document.all
        2. 3.1.1.2. [建议] 对于多个元素的集合，尽可能使用 context.getElementsByTagName 获取。其中 context 可以为 document 或其他元素。指定 tagName 参数为 * 可以获得所有子元素
        3. 3.1.1.3. [建议] 遍历元素集合时，尽量缓存集合长度。如需多次操作同一集合，则应将集合转为数组
        4. 3.1.1.4. [建议] 获取元素的直接子元素时使用 children。避免使用childNodes，除非预期是需要包含文本、注释和属性类型的节点
      2. 3.1.2. 样式获取
        1. 3.1.2.1. [建议] 获取元素实际样式信息时，应使用 getComputedStyle 或 currentStyle
      3. 3.1.3. 样式设置
        1. 3.1.3.1. [建议] 尽可能通过为元素添加预定义的 className 来改变元素样式，避免直接操作 style 设置
        2. 3.1.3.2. [强制] 通过 style 对象设置元素样式时，对于带单位非 0 值的属性，不允许省略单位
      4. 3.1.4. DOM 操作
        1. 3.1.4.1. [建议] 操作 DOM 时，尽量减少页面 reflow
        2. 3.1.4.2. [建议] 尽量减少 DOM 操作
      5. 3.1.5. DOM 事件
        1. 3.1.5.1. [建议] 优先使用 addEventListener / attachEvent 绑定事件，避免直接在 HTML 属性中或 DOM 的 expando 属性绑定事件处理
        2. 3.1.5.2. [建议] 使用 addEventListener 时第三个参数使用 false
        3. 3.1.5.3. [建议] 在没有事件自动管理的框架支持下，应持有监听器函数的引用，在适当时候（元素释放、页面卸载等）移除添加的监听器

    开发者信息能够体现开发人员对文件的贡献，并且能够让遇到问题或希望了解相关信息的人找到维护人。通常情况文件在被创建时标识的是创建者。随着项目的进展，越来越多的人加入，参与这个文件的开发，新的作者应该被加入 @author 标识。

    解释：

    在 JavaScript 中，无需特别的关键词就可以使用闭包，一个函数可以任意访问在其定义的作用域外的变量。需要注意的是，函数的作用域是静态的，即在定义时决定，与调用的时机和方式没有任何关系。

    闭包会阻止一些变量的垃圾回收，对于较老旧的 JavaScript 引擎，可能导致外部所有变量均无法回收。

    首先一个较为明确的结论是，以下内容会影响到闭包内变量的回收：

    • 嵌套的函数中是否有使用该变量。
    • 嵌套的函数中是否有 直接调用eval。
    • 是否使用了 with 表达式。

    Chakra、V8 和 SpiderMonkey 将受以上因素的影响，表现出不尽相同又较为相似的回收策略，而 JScript.dll 和 Carakan 则完全没有这方面的优化，会完整保留整个 LexicalEnvironment 中的所有变量绑定，造成一定的内存消耗。

    由于对闭包内变量有回收优化策略的 Chakra、V8 和 SpiderMonkey 引擎的行为较为相似，因此可以总结如下，当返回一个函数 fn 时：

    1. 如果 fn 的 [[Scope]] 是 ObjectEnvironment（with 表达式生成 ObjectEnvironment，函数和 catch 表达式生成 DeclarativeEnvironment），则：
      1. 如果是 V8 引擎，则退出全过程。
      2. 如果是 SpiderMonkey，则处理该 ObjectEnvironment 的外层 LexicalEnvironment。
    2. 获取当前 LexicalEnvironment 下的所有类型为 Function 的对象，对于每一个 Function 对象，分析其 FunctionBody：
      1. 如果 FunctionBody 中含有 直接调用 eval，则退出全过程。
      2. 否则得到所有的 Identifier。
      3. 对于每一个 Identifier，设其为 name，根据查找变量引用的规则，从 LexicalEnvironment 中找出名称为 name 的绑定 binding。
      4. 对 binding 添加 notSwap 属性，其值为 true。
    3. 检查当前 LexicalEnvironment 中的每一个变量绑定，如果该绑定有 notSwap 属性且值为 true，则：
      1. 如果是 V8 引擎，删除该绑定。
      2. 如果是 SpiderMonkey，将该绑定的值设为 undefined，将删除 notSwap 属性。

    对于 Chakra 引擎，暂无法得知是按 V8 的模式还是按 SpiderMonkey 的模式进行。

    如果有 非常庞大 的对象，且预计会在 老旧的引擎 中执行，则使用闭包时，注意将闭包不需要的对象置为空引用。

    DOM 操作

    [建议] 操作 DOM 时，尽量减少页面 reflow

    解释：

    页面 reflow 是非常耗时的行为，非常容易导致性能瓶颈。下面一些场景会触发浏览器的reflow：

    • DOM元素的添加、修改（内容）、删除。
    • 应用新的样式或者修改任何影响元素布局的属性。
    • Resize浏览器窗口、滚动页面。
    • 读取元素的某些属性（offsetLeft、offsetTop、offsetHeight、offsetWidth、scrollTop/Left/Width/Height、clientTop/Left/Width/Height、getComputedStyle()、currentStyle(in IE)) 。
    [建议] 尽量减少 DOM 操作

    解释：

    DOM 操作也是非常耗时的一种操作，减少 DOM 操作有助于提高性能。举一个简单的例子，构建一个列表。我们可以用两种方式：

    1. 在循环体中 createElement 并 append 到父元素中。
    2. 在循环体中拼接 HTML 字符串，循环结束后写父元素的 innerHTML。

    第一种方法看起来比较标准，但是每次循环都会对 DOM 进行操作，性能极低。在这里推荐使用第二种方法。

    DOM 事件

    [建议] 优先使用 addEventListener / attachEvent 绑定事件，避免直接在 HTML 属性中或 DOM 的 expando 属性绑定事件处理

    解释：

    expando 属性绑定事件容易导致互相覆盖。

    [建议] 使用 addEventListener 时第三个参数使用 false

    解释：

    标准浏览器中的 addEventListener 可以通过第三个参数指定两种时间触发模型：冒泡和捕获。而 IE 的 attachEvent 仅支持冒泡的事件触发。所以为了保持一致性，通常 addEventListener 的第三个参数都为 false。

    [建议] 在没有事件自动管理的框架支持下，应持有监听器函数的引用，在适当时候（元素释放、页面卸载等）移除添加的监听器
